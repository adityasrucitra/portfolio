Maritime Vessel Tracking Platform

Context

Shipping company with 50+ vessels operating across Southeast Asia and Middle East. Vessels operate in areas with variable connectivity (LTE, satellite, port connections). Operations team based in Jakarta with 24/7 monitoring requirement.

Problem

Manual position reporting via radio/SMS; no real-time visibility; unable to optimize routes or respond to emergencies quickly. Paper-based fuel and maintenance logs created compliance risk.

Constraints

  • Network latency: 500ms–2s depending on connection type
  • Satellite connectivity: High cost, low bandwidth (pay-per-MB)
  • Vessel equipment: Legacy GPRS modems, limited processing power
  • Operational requirement: Position update every 15 minutes during transit

System Architecture

GPS Device (Garmin, u-blox) → Edge Gateway (Raspberry Pi w/ MQTT) → AWS IoT Core → Backend API (PHP/CodeIgniter) → MySQL (operational data) + TimescaleDB (time-series) → Web Dashboard (real-time map)

Key Engineering Decisions

  • MQTT over HTTP: Lower bandwidth, persistent connection, QoS guarantees for offline-first sync
  • AWS IoT Core: Managed MQTT broker, built-in authentication, scales to thousands of devices
  • Edge buffering: Local SQLite caching for 24h of position data; sync when connection available
  • TimescaleDB: Time-series data optimized for position history queries; 10x compression vs raw MySQL

Implementation Details

  • Data ingestion: 50 vessels × 4 updates/hour = ~8k messages/day; each ~200 bytes
  • Processing pipeline: AWS Lambda functions trigger on MQTT topic → validate → geocode → store in TimescaleDB → publish to WebSocket for live dashboard
  • Storage: Position archive partitioned by month; 12 months = ~500MB; geospatial indexes for proximity queries

Failure Handling & Resilience

  • Offline mode: Edge gateway buffers up to 10k messages; syncs when LTE/satellite reconnects
  • Retry strategy: MQTT QoS 1 (at-least-once); Lambda DLQ for failed processing; manual replay via admin API
  • Satellite failover: LTE primary; automatic fallback to Iridium satellite if no LTE for 5 minutes
  • Message deduplication: Using MQTT message ID + timestamp; prevents duplicate position reports

Impact & Outcomes

  • Real-time visibility: Operations team now has live position, fuel level, engine status for all 50 vessels
  • Route optimization: Saved ~8% fuel consumption via optimized routing based on actual weather/traffic data
  • Compliance: Automated position logging for regulatory reporting; eliminated manual data entry
  • Uptime: 99.7% dashboard availability over 18 months; zero data loss on vessels

Industrial Fuel Monitoring & Analytics

Context

Marine fleet operator with 12 refueling stations across 4 countries. Monthly fuel consumption: ~500,000 liters. Fuel cost: 40% of operating budget. Manual reconciliation between supply, consumption, and billing occurs monthly—typically 3–5 discrepancies.

Problem

Manual fuel reconciliation is slow (1 week per month), error-prone (±5% variance), and prevents real-time visibility into consumption patterns. Inability to detect theft, leaks, or meter calibration drift until end of month.

Constraints

  • Legacy infrastructure: Fuel pump meters not directly connected; manual gauge readings
  • Budget: Cannot replace all fuel dispensers; must integrate with existing hardware
  • Data latency: Readings from remote sites (up to 8 hours delay over satellite)
  • Accuracy requirement: ±0.5% variance; detect 5L+ discrepancies in real-time

System Architecture

Flow meter sensors (Omega, Emerson) → IoT edge gateway (LoRaWAN) → AWS IoT Core → Time-series database → Anomaly detection pipeline (Python/Lambda) → PostgreSQL (clean data) → Tableau dashboard + alerts

Key Engineering Decisions

  • LoRaWAN over WiFi/LTE: Low-power wide-area network; 10km+ range; one gateway per site
  • Time-series database (InfluxDB): Optimized for continuous metric streams; 10x faster queries than row-oriented databases
  • Anomaly detection via ML: Isolation Forest algorithm detects unusual consumption patterns; alerts on theft/leak risk
  • Data reconciliation layer: Weighted average algorithm combines meter readings + manual counts; produces single source of truth

Implementation Details

  • Ingestion: 12 sites × 4 readings/hour = 1.1k data points/day; each ~50 bytes → 55KB/day
  • Processing: Real-time aggregation (hourly summaries) + batch anomaly detection (nightly); historical reconciliation (monthly)
  • Storage: InfluxDB with 2-year retention; PostgreSQL for reconciliation audit trail; S3 for monthly export reports

Failure Handling & Resilience

  • Sensor failure: Edge gateway maintains local queue; if sensor offline >1 hour, SMS alert to site manager
  • Network outage: Data cached locally for 72 hours; automatic sync when connectivity restored
  • Data drift correction: Monthly manual verification; drift >2% triggers calibration alert
  • Audit trail: Every reconciliation decision logged with timestamp + username; immutable via blockchain hash

Impact & Outcomes

  • Reconciliation time: Reduced from 1 week to 2 hours; monthly fuel variance now <0.3%
  • Loss prevention: Detected 2 major fuel theft incidents (400L, 600L) that would have gone unnoticed
  • Cost optimization: Identified consumption anomalies; led to fleet maintenance improvements saving ~3% fuel
  • Regulatory compliance: Complete audit trail for fuel cost tracking and carbon reporting

Remote Industrial IoT Gateway with Failover Architecture

Context

Mining operation in remote Indonesia with 30+ sensors (temperature, vibration, power, water levels). Site located 150km from nearest cellular tower; LTE coverage: 40%; satellite internet: expensive ($500/GB). Operations staff: 20 engineers; mission-critical safety systems require continuous telemetry.

Problem

Unstable LTE connectivity causes telemetry gaps; missing data prevents early detection of equipment failures (vibration spikes = bearing damage). Satellite Internet is unreliable and expensive. Current solution: Manual daily data collection via USB drive—24–48 hour reporting lag.

Constraints

  • Network: LTE availability 30–60% of time; latency 800ms–3s when available
  • Satellite: 1 Mbps shared uplink; $500/10GB overage charges
  • Power: Solar + diesel backup; 10kW peak, 500W baseline continuous
  • Bandwidth budget: <50MB/day including all telemetry + system updates

System Architecture

30 sensors (Modbus/RS-485) → Edge gateway (Intel NUC, Ubuntu) → Local processing → LTE modem (primary) + Iridium satellite (failover) → AWS IoT Core → Kinesis stream → Lambda + S3 → Real-time dashboard + alerts

Key Engineering Decisions

  • Edge-first processing: All data aggregation happens locally; only summaries (not raw samples) sent to cloud
  • Intelligent failover: LTE monitored every 30s; automatic switch to Iridium if latency >2s or packet loss >5%
  • Compression & delta encoding: Only changed values transmitted; 95% bandwidth reduction vs raw telemetry
  • Offline-first design: Critical alerts (vibration spike, temperature >80°C) stored locally + trigger on-site alarm; synced when online

Implementation Details

  • Data collection: 30 sensors sampled every 10s → 1-minute rolling averages (min/max/mean) → uploaded every 5 minutes
  • Storage: 14-day local SQLite buffer (128MB); syncs to AWS S3 + TimescaleDB on connection recovery
  • Processing: Critical alerts evaluated in-gateway (hardcoded rules); anomalies sent to cloud for ML scoring

Failure Handling & Resilience

  • Gateway failure: Dual gateway setup; automatic failover via VRRP protocol; shared storage on NAS
  • Sensor failure: Missing data flagged; threshold-based alerting (e.g., "no vibration data for 30 min = sensor offline")
  • Cloud API failure: Gateway retries with exponential backoff; stores data locally for 2 weeks
  • Power loss: UPS keeps gateway online for 4 hours; graceful shutdown to prevent data corruption

Impact & Outcomes

  • Data availability: Improved from 40% (manual) to 99.2% (automated with edge buffering)
  • Incident detection: Vibration anomalies now detected within 15 minutes vs 24 hours; prevented 3 bearing failures
  • Cost savings: Bandwidth usage: 35MB/day (vs projected 500MB for raw telemetry); satellite overage charges eliminated
  • Operational efficiency: Site engineers can now troubleshoot remotely via telemetry; reduced on-site visits by 40%

Technology Decisions & Tradeoffs

How we choose between competing technologies based on operational constraints.

MQTT Over HTTP Polling

For real-time IoT telemetry from resource-constrained devices, MQTT (publish-subscribe) provides persistent connections, QoS guarantees (at-least-once delivery), and automatic message queuing during network outages. HTTP polling requires the device to repeatedly ask for updates, consuming 5–10x more bandwidth and processing power. Trade-off: MQTT requires a broker (AWS IoT Core), but eliminates the complexity of managing reconnection logic on each device.

AWS IoT Core vs Self-Hosted MQTT Broker

Self-hosted brokers (Mosquitto, HiveMQ) offer full control but require: infrastructure scaling, certificate management, DDoS protection, and 24/7 monitoring. AWS IoT Core is a managed service with: built-in device authentication (X.509), automatic scaling to millions of connections, lifecycle management, and rule engine for filtering/routing. For mission-critical maritime systems, the managed service eliminates operational burden. Cost: ~$0.04 per million MQTT messages vs ~$500/month for self-hosted infrastructure.

TimescaleDB Over Traditional MySQL

Time-series data (vessel positions, sensor readings) has different query patterns than transactional data: append-heavy, rarely updated, and optimized for range queries. TimescaleDB (PostgreSQL extension) compresses time-series data 10–15x better than row-oriented storage, and range queries (e.g., "all position points in the last 24 hours") are 5–100x faster. Trade-off: TimescaleDB is specialized; we use PostgreSQL for operational data (user accounts, transactions) and TimescaleDB for analytics.

Edge Processing vs Cloud-Only

Cloud-only processing requires all raw sensor data to reach the cloud, which in remote environments means massive bandwidth usage (500MB+ per site per day). Edge processing (local aggregation) reduces this to 5–10% of the original size through: minute-level rolling averages, anomaly detection (outlier rejection), and delta encoding (only send changed values). Critical alerts (vibration spike, temperature exceeding threshold) trigger immediately on-edge without waiting for cloud round-trip. Trade-off: Increased device complexity, but massive bandwidth savings and lower operational cost.

LoRaWAN Over WiFi/LTE

For stationary industrial sensors at geographically dispersed sites, LoRaWAN (Long Range Wide Area Network) provides: 10km+ range, low power consumption (battery life: years), and per-gateway cost (~$300/unit). WiFi requires each sensor to be within 50m and managed individually. LTE requires cellular coverage (often unavailable in remote areas) and monthly per-device connectivity fees. LoRaWAN aggregates multiple sensors through one gateway, reducing infrastructure cost and operational complexity. Trade-off: Lower bandwidth (best for periodic readings, not streaming).

Compression & Delta Encoding

Transmitting raw sensor samples every 10 seconds from 30 sensors = 2.6GB per day. Aggregating to 1-minute rolling averages (min/max/mean) reduces this to 26MB. Adding delta encoding (only transmit values that changed >threshold) reduces further to 5MB. For satellite connectivity ($500/10GB overage), this difference is operational necessity: 26MB/day is $13/month; 5MB/day is $2/month. Trade-off: Slight data loss on non-critical metrics (e.g., fuel level that didn't change), but critical metrics (vibration, temperature) are never delta-encoded.

Offline-First Architecture

Traditional systems fail when network is unavailable. Offline-first design: all data stored locally first, synced to cloud when connectivity available. Benefits: uninterrupted monitoring, zero data loss, and graceful degradation. Implementation: local SQLite buffer on gateway stores up to 14 days of telemetry; critical alerts trigger on-gateway rules (hardcoded thresholds); non-critical anomalies queued for cloud ML scoring when online. Trade-off: increased local storage (128MB per gateway) and added complexity, but ensures mission-critical systems never go blind.

Production-Grade Resilience

Real-world failure scenarios from production systems and how they're prevented or recovered from.

Network Outage: Satellite Failover Protocol

Maritime operations require continuous position updates even when LTE fails. Our system automatically detects when LTE latency exceeds 2 seconds or packet loss exceeds 5% for 30 seconds, then switches to Iridium satellite connectivity.

Scenario: A vessel loses LTE signal entering international waters. System detects outage, queued messages are flushed over satellite link (higher cost, lower bandwidth, but uninterrupted monitoring).
Mitigation: Hardcoded latency/packet-loss thresholds trigger automatic failover. Critical alerts (collision risk, fire) are sent via both channels. Cost impact: extra $50/month per vessel, but prevents catastrophic blind spots.

Device Failure: Edge Gateway Redundancy

A single gateway in a remote mining site collects data from 30+ sensors. If the gateway crashes, all telemetry is lost. We deploy dual gateways with automatic failover using VRRP protocol (Virtual Router Redundancy).

Scenario: Primary gateway power-fails during heavy rain. Backup gateway detects missing heartbeat within 3 seconds and takes over. Sensors switch to backup gateway automatically via ARP announcement.
Mitigation: Dual NUC setup with shared NAS storage (24h local buffering). UPS provides 4-hour hold time for graceful degradation. Total cost: ~$2k extra, but eliminates single point of failure for mission-critical monitoring.

Data Loss: Offline-First Buffering

Cloud APIs fail. Satellite connectivity drops. Internet storms cause packet loss. Traditional systems lose data in these scenarios. Our offshore systems buffer all data locally first, sync when healthy.

Scenario: AWS IoT Core becomes unreachable for 8 hours due to DDoS. Local SQLite buffer on every edge gateway stores 14 days of raw telemetry. When connectivity restores, queued messages replay in order, zero data loss.
Mitigation: All data written to local SQLite before cloud transmission. Message IDs + timestamps enable deduplication on recovery. Critical alerts (vibration spike) trigger local hardcoded rules immediately, don't wait for cloud scoring. Extra local storage: 128MB per gateway.

Data Corruption: Graceful Degradation

Sensor reads garbage data. Network corrupts packets. Database constraints are violated. Corrupted data cascades through systems. We validate early and degrade gracefully rather than crash.

Scenario: Temperature sensor returns -999°C (sensor malfunction). System flags the data point, logs to audit trail, but does NOT block the pipeline. Dashboard shows the reading with a warning badge. Alert only triggers if pattern repeats (outlier vs sensor death).
Mitigation: Multi-layer validation: edge gateway validates ranges (temp -40 to 150°C), backend validates patterns (isolation forest anomaly detection), database has CHECK constraints. Failed validation = quarantine + audit log, never crash. Operators see exactly what data is suspect.
